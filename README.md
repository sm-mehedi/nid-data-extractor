# Bangladesh NID Extractor

A REST API that accepts photos of the front and back of a Bangladesh National
ID (NID) card and returns structured, English-translated JSON: name, parents'
names, date of birth, NID number, and address.

## Project overview

The service runs a 6-stage pipeline per request:

1. **Upload** — `front_image` + `back_image`, multipart, JPG/JPEG/PNG.
2. **Cheap local checks** (no API cost) — extension/size/decodability, blur
   (Laplacian variance), exposure (brightness histogram), glare (localized
   bright-pixel blob), and card-boundary detection (auto-crop + deskew if a
   clean quad is found; reject if the card's own edge is cut off in the
   photo).
3. **Google Cloud Vision OCR** — `DOCUMENT_TEXT_DETECTION` on both images
   (Bengali + English), including the back's machine-readable zone (MRZ).
4. **MRZ parse + checksum validation** — ICAO 9303 TD1 format (3 lines × 30
   chars), per-field checksums, cross-checked against the front-of-card data.
5. **Gemini** (structuring + translation) — reads both images (with the OCR
   text as a hint), translates Bengali fields meaning-first (not
   word-for-word), and makes the holistic "is this really an NID" judgment.
6. **Merge + respond** — MRZ-verified fields (checksum-passed) take priority
   over Gemini's own reading when the two disagree.

**Why two AI services instead of one:** Cloud Vision is Google's officially
"Supported" (not experimental) OCR engine for printed Bengali script — the
best tool for *reading* it reliably. Gemini is better at *understanding and
translating with preserved meaning* and at making a holistic plausibility
call. Splitting the two problems across the tool best suited to each is more
robust than asking one model to do both.

## Technology stack

- **Language / framework**: Python 3.11, FastAPI, Uvicorn
- **Image processing**: OpenCV (`opencv-python-headless`), NumPy, Pillow
- **OCR**: Google Cloud Vision API (`DOCUMENT_TEXT_DETECTION`)
- **LLM (structuring/translation)**: Google Gemini (`google-generativeai`)
- **Rate limiting**: `slowapi`
- **Testing**: `pytest`, `pytest-mock`, synthetic images + mocked API responses
- **Packaging**: Docker / docker-compose

## AI/OCR libraries used

- `google-cloud-vision` — Cloud Vision REST/client for OCR
- `google-generativeai` — Gemini for translation + structuring + plausibility judgment
- `opencv-python-headless` — blur/exposure/glare/card-boundary detection, auto-crop + deskew

## Two deliverable variants, one codebase

- **API-only** (`nid-extractor-api-only.zip`): the exact assignment
  deliverable — `app/`, Dockerfile, `docker-compose.yml`, tests, docs. No
  `frontend/` folder.
- **With frontend** (`nid-extractor-with-frontend.zip`): everything above,
  plus a small bonus `frontend/` (plain HTML/CSS/JS, no framework) that calls
  the same API via `fetch()` on the same origin. `app/main.py` mounts it at
  `GET /` **only if the `frontend/` directory is present** — so the exact
  same source tree serves both variants; nothing is edited between them.

## Build instructions

```bash
docker build -t nid-extractor .
```

## Run instructions

1. Copy `.env.example` to `.env` and fill in real credentials:
   ```bash
   cp .env.example .env
   ```
   You need **either** `GOOGLE_CLOUD_VISION_API_KEY` **or**
   `GOOGLE_APPLICATION_CREDENTIALS` (path to a service-account JSON) for OCR,
   and `GEMINI_API_KEY` for structuring/translation.

2. Run with Docker Compose (API only):
   ```bash
   docker compose up --build
   ```
   Or, for the bonus frontend variant (only meaningful if `frontend/` exists
   in this copy of the repo):
   ```bash
   docker compose -f docker-compose.web.yml up --build
   ```

3. Or run directly with `docker run`:
   ```bash
   docker build -t nid-extractor .
   docker run --env-file .env -p 8000:8000 nid-extractor
   ```

4. Or run locally without Docker:
   ```bash
   python -m venv .venv
   .venv/Scripts/activate        # Windows
   pip install -r requirements-dev.txt
   uvicorn app.main:app --reload
   ```

The API listens on `$PORT` (defaults to 8000; Railway injects this
automatically in production — do not set it manually there).

## API endpoint(s)

- `POST /api/v1/nid/extract` — multipart form fields `front_image`,
  `back_image` → structured JSON (see [API_DOCS.md](API_DOCS.md) for the full
  contract, error table, and examples).
- `GET /health` — liveness check.
- `GET /docs` — interactive Swagger UI (auto-generated by FastAPI).

## Security

- Per-IP rate limiting (`RATE_LIMIT_PER_MINUTE`, default 8/min) and a
  concurrency cap (`CONCURRENCY_LIMIT`) protect the Cloud Vision/Gemini
  free-tier quota.
- Upload size is enforced **while streaming in**, not after buffering the
  whole body.
- Uploaded images and extracted personal data are **never persisted** to
  disk or a database — everything is in-memory, per-request, discarded after
  the response.
- Logs record success/failure + reason only — never full images or
  extracted personal fields.
- Standard security headers (`X-Content-Type-Options`, `X-Frame-Options`,
  `Content-Security-Policy`, `Referrer-Policy`) on every response.
- Optional shared-secret header (`API_SHARED_SECRET` / `X-API-Key`) to lock
  the endpoint down if it shouldn't be fully public.
- TLS is handled by Railway's edge in production — nothing to configure here.

## A note on the address fields

Real Bangladesh NID cards print a single ঠিকানা (address), not separate
present/permanent fields. Both `presentAddress` and `permanentAddress` in the
response carry that same value — this is expected behavior on a real card,
not a bug, and isn't repeated as a per-response warning (it would fire on
almost every request and stop being a useful signal). `warnings` is reserved
for things unusual to *that specific* upload: an unreadable field, or a
front/back MRZ mismatch.

## Explicitly out of scope

This system verifies **internal consistency** only — MRZ checksums,
front/back cross-checks, and an AI plausibility judgment. It does **not**:
performing forgery/tamper detection, check against Bangladesh's actual NID
registry, or match the card photo against a live selfie. It is also not
designed for high-concurrency production load (that would need a task
queue, horizontal scaling, and paid API tiers).
