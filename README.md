# Bangladesh NID Extractor

A REST API that accepts photos of the front and back of a Bangladesh National
ID (NID) card and returns structured, English-translated JSON: name, parents'
names, date of birth, NID number, and address.

## Project overview

The service runs a 6-stage pipeline per request:

1. **Upload** — `front_image` + `back_image`, multipart, JPG/JPEG/PNG.
2. **Cheap local checks** (no API cost) — extension/size/decodability
   (hard rejection), blur (Laplacian variance), exposure (brightness
   histogram), and glare (localized bright-pixel blob) are soft checks that
   add a warning and let the request continue rather than blocking it (real
   photos — especially recompressed by messaging apps, or a laminated card
   catching some glare — routinely trip these thresholds while still being
   legible). Card-boundary detection auto-crops + deskews if a clean quad is
   found, and is the one hard rejection in this stage: if the card's own
   edge is cut off in the photo, that's missing data no processing can
   recover.
3. **Google Cloud Vision OCR** — `DOCUMENT_TEXT_DETECTION` on both images
   (Bengali + English), including the back's machine-readable zone (MRZ).
4. **MRZ parse + checksum validation** — ICAO 9303 TD1 format (3 lines × 30
   chars), per-field checksums, cross-checked against the front-of-card data.
5. **Gemini** (structuring + translation) — reads both images (with the OCR
   text as a hint), translates Bengali fields meaning-first (not
   word-for-word), and makes the holistic "is this really an NID" judgment.
6. **Merge + respond** — MRZ-verified fields (checksum-passed) take priority
   over Gemini's own reading when the two disagree.

**Why two AI services instead of one:** Cloud Vision is Google's officially
"Supported" (not experimental) OCR engine for printed Bengali script — the
best tool for *reading* it reliably. Gemini is better at *understanding and
translating with preserved meaning* and at making a holistic plausibility
call. Splitting the two problems across the tool best suited to each is more
robust than asking one model to do both.

## Technology stack

- **Language / framework**: Python 3.11, FastAPI, Uvicorn
- **Image processing**: OpenCV (`opencv-python-headless`), NumPy, Pillow
- **OCR**: Google Cloud Vision API (`DOCUMENT_TEXT_DETECTION`)
- **LLM (structuring/translation)**: Google Gemini (`google-genai`)
- **Rate limiting**: `slowapi`
- **Testing**: `pytest`, `pytest-mock`, synthetic images + mocked API responses
- **Packaging**: Docker / docker-compose

## AI/OCR libraries used

- `google-cloud-vision` — Cloud Vision REST/client for OCR
- `google-genai` — Gemini for translation + structuring + plausibility judgment (the unified SDK; the older `google-generativeai` package has no `thinking_config` support and is fully deprecated)
- `opencv-python-headless` — blur/exposure/glare/card-boundary detection, auto-crop + deskew

## Two deliverable variants, one codebase

- **API-only** (`nid-extractor-api-only.zip`): the exact assignment
  deliverable — `app/`, Dockerfile, `docker-compose.yml`, tests, docs. No
  `frontend/` folder.
- **With frontend** (`nid-extractor-with-frontend.zip`): everything above,
  plus a small bonus `frontend/` (plain HTML/CSS/JS, no framework) that calls
  the same API via `fetch()` on the same origin. `app/main.py` mounts it at
  `GET /` **only if the `frontend/` directory is present** — so the exact
  same source tree serves both variants; nothing is edited between them.

## Build instructions

```bash
docker build -t nid-extractor .
```

## Run instructions

1. Copy `.env.example` to `.env` and fill in real credentials:
   ```bash
   cp .env.example .env
   ```
   You need **either** `GOOGLE_CLOUD_VISION_API_KEY` **or**
   `GOOGLE_APPLICATION_CREDENTIALS` (path to a service-account JSON) for OCR,
   and `GEMINI_API_KEY` for structuring/translation.

2. Run with Docker Compose (API only):
   ```bash
   docker compose up --build
   ```
   Or, for the bonus frontend variant (only meaningful if `frontend/` exists
   in this copy of the repo):
   ```bash
   docker compose -f docker-compose.web.yml up --build
   ```

3. Or run directly with `docker run`:
   ```bash
   docker build -t nid-extractor .
   docker run --env-file .env -p 8000:8000 nid-extractor
   ```

4. Or run locally without Docker:
   ```bash
   python -m venv .venv
   .venv/Scripts/activate        # Windows
   pip install -r requirements-dev.txt
   uvicorn app.main:app --reload
   ```

The API listens on `$PORT` (defaults to 8000; Railway injects this
automatically in production — do not set it manually there).

## API endpoint(s)

- `POST /api/v1/nid/extract` — multipart form fields `front_image`,
  `back_image` → structured JSON (see [API_DOCS.md](API_DOCS.md) for the full

- `GET /health` — liveness check.
- `GET /docs` — interactive Swagger UI (auto-generated by FastAPI).

## Security

- Per-IP rate limiting (`RATE_LIMIT_PER_MINUTE`, default 8/min) and a
  concurrency cap (`CONCURRENCY_LIMIT`) protect the Cloud Vision/Gemini
  free-tier quota. Behind a proxy (Cloud Run, or any other platform that
  terminates the connection before it reaches this container), the real
  end-user IP only reaches `request.client.host` because both the
  Dockerfile's `--proxy-headers --forwarded-allow-ips='*'` and an explicit
  `ProxyHeadersMiddleware` in `app/main.py` trust `X-Forwarded-For`. Without
  either, every request looks like it came from the proxy itself, and the
  "per-IP" limit silently becomes one shared bucket for all real users.
- Upload size is enforced **while streaming in**, not after buffering the
  whole body.
- Uploaded images and extracted personal data are **never persisted** to
  disk or a database — everything is in-memory, per-request, discarded after
  the response.
- Logs record success/failure + reason only — never full images or
  extracted personal fields.
- Standard security headers (`X-Content-Type-Options`, `X-Frame-Options`,
  `Content-Security-Policy`, `Referrer-Policy`) on every response.
- Optional shared-secret header (`API_SHARED_SECRET` / `X-API-Key`) to lock
  the endpoint down if it shouldn't be fully public.
- TLS is handled by Railway's edge in production — nothing to configure here.

## Reliability & timeouts

- Gemini calls use `GEMINI_TIMEOUT_SECONDS` (default 100s) — two-image
  structured extraction with translation is a heavier request than a plain
  text prompt, so this is deliberately generous.
- `thinking_config` is set to a zero thinking budget for thinking-capable
  models (this is structured extraction, not a task that benefits from
  extended reasoning) — cuts both latency and cost. If the configured model
  doesn't support `thinking_config` at all, the call transparently retries
  once without it rather than failing.
- `GEMINI_RETRY_BACKOFF_SECONDS` (default `3,8,15`) retries with escalating
  backoff specifically for 429 (rate limit), 503 (model overloaded), and
  timeout/504 (deadline exceeded) — the failure modes actually worth a
  second attempt. Other errors (auth, bad request, safety block) fail
  immediately. Each number is a real, separate wait between attempts — the
  default schedule is 3 retries (4 total attempts), confirmed by a test that
  asserts the exact sleep sequence used, not just that a retry happened once.
- `GEMINI_FALLBACK_MODEL` (default: auto-derived as `<GEMINI_MODEL>-lite`) —
  if the primary model exhausts its entire retry schedule and the final
  error is still a 503 (overloaded), one extra attempt is made against the
  fallback model before giving up. Different model variants often have
  separate capacity pools. No retries on the fallback attempt itself.
- **If you deploy behind a platform with its own request timeout (e.g.
  Cloud Run's default 300s), make sure it stays comfortably above the worst
  case here.** With the defaults: the primary model can take up to
  `4 attempts × 100s = 400s` plus `3+8+15 = 26s` of backoff = **426s**, and
  if it ends on a 503 there's one more fallback attempt (+100s) = **526s**,
  before the Cloud Vision OCR stage (~60s, unchanged) is even added —
  **~586s worst case total.** That is well past a 300s platform timeout.
  Raise the platform's request timeout to comfortably exceed this (Cloud Run
  supports up to 3600s), or shorten `GEMINI_TIMEOUT_SECONDS` /
  `GEMINI_RETRY_BACKOFF_SECONDS` to fit an existing budget.

## A note on the address fields

Real Bangladesh NID cards print a single ঠিকানা (address), not separate
present/permanent fields. Both `presentAddress` and `permanentAddress` in the
response carry that same value. Gemini doesn't reliably mirror this into
both output fields on every call — it's been observed returning one field
as `null` with a low-confidence flag on one run and correctly mirroring on
another run, for the exact same card design — so this is enforced
deterministically in code (`app/services/pipeline.py`, `_mirror_addresses`),
not left to the model: if either field comes back empty and the other has a
value, the value is copied into the empty one. When both fields already
match (the expected case), this is silent — no warning, since firing on
almost every request would stop being a useful signal. A warning only
appears when the mirroring actually had to fill a gap, which is exactly the
case worth flagging.

## A note on name translation reliability

Gemini's Bengali-to-English translation of `name`/`fatherName`/`motherName`
has been observed to be inconsistent *within a single response* — e.g.
`presentAddress` translated correctly while `fatherName` came back as raw,
untranslated Bengali script in the same call. Rather than trust the model to
translate every field correctly every time, each name field is checked for
Bengali Unicode characters (U+0980–U+09FF) after the response comes back
(`app/services/pipeline.py`, `_ensure_translated`). Any field that still
contains Bengali script gets a local, offline phonetic transliteration
fallback (`app/services/translit.py`, via `indic-transliteration`) instead
of being returned untranslated, with a warning noting the fallback was used.
This fallback is a hard guarantee against untranslated Bengali script
reaching the response — it is **not** a quality-equivalent substitute for
Gemini's own translation. It runs in two ordered stages:

1. **Recognized prefixes/titles are matched and replaced directly** —
   Md., Mst., Syed, Sheikh, Alhaj, Haji, and Late (a status marker meaning
   the named person, typically a parent, is deceased — not part of the name
   itself) are never phonetically transliterated. These can stack (e.g.
   "মৃত মোঃ..." -> "Late Md...."), so recognized prefixes are stripped
   in sequence from the front of the string before anything else happens.
2. **Whatever remains is phonetically transliterated** via
   `indic-transliteration`. Two systematic corrections are applied, both
   because the library applies Sanskrit/Devanagari Romanization conventions
   to Bengali text rather than Bengali-specific ones (confirmed by testing
   every scheme the library offers, not by assumption):
   - ব is corrected from "v" to "b" (e.g. "Veoya" -> "Beoya") — safe as a
     blanket substitution, since the only other letter that could plausibly
     produce a "v" sound, ভ (bha), always transliterates to "bh" in this
     library, never a bare "v".
   - Inherent vowels Bengali pronunciation drops (schwa deletion) are
     **not** corrected — e.g. "Jamsed" still comes out as "Jamaseda". No
     scheme this library offers accounts for Bengali's schwa-deletion rules,
     and building a custom rule set for it risks introducing new errors
     without a clear payoff, so this is documented as a known, unfixed
     limitation rather than patched with something fragile.

   The library also doesn't reliably transliterate every character on its
   own (some conjunct/nukta combinations pass straight through untouched),
   so a final pass strips anything that survives — the one strict guarantee
   is no Bengali script in the output, not phonetic accuracy.

It exists purely as a safety net for the rare case the primary path fails,
not as an alternate primary path.

## Explicitly out of scope

This system verifies **internal consistency** only — MRZ checksums,
front/back cross-checks, and an AI plausibility judgment. It does **not**:
performing forgery/tamper detection, check against Bangladesh's actual NID
registry, or match the card photo against a live selfie. It is also not
designed for high-concurrency production load (that would need a task
queue, horizontal scaling, and paid API tiers).
